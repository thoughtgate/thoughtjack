## 1. Context & Decision Rationale

This specification defines ThoughtJack's built-in scenario system — OATF attack scenarios auto-discovered at build time from the official OATF scenario library, embedded in the binary for zero-configuration usage.

### 1.1 Motivation

ThoughtJack's YAML-driven approach is powerful but creates an onboarding barrier. Built-in scenarios solve this:

```bash
# With built-in scenarios
thoughtjack scenarios run oatf-002
```

### 1.2 Design Principles

| Principle | Rationale |
|-----------|-----------|
| **Zero dependencies** | Binary is self-contained — no library checkout, no network, no file system |
| **Auto-discovery** | Adding scenarios to the OATF library automatically embeds them — no Rust code changes |
| **Validation at build time** | Only OATF documents that pass `oatf::load()` validation are embedded |
| **Discoverable** | Users can list, filter, and inspect built-in scenarios from the CLI |
| **Composable** | Built-in scenarios work with all existing CLI flags |
| **Representative** | Scenarios cover the OATF attack taxonomy across protocols |
| **Educational** | Viewing a scenario's YAML teaches the OATF document format by example |

### 1.3 Relationship to Other Specs

| Spec | Relationship |
|------|--------------|
| TJ-SPEC-013 (OATF Integration) | Built-in scenarios are valid OATF documents parsed by the SDK |
| TJ-SPEC-007 (CLI Interface) | `scenarios` subcommand and `--scenario` flag on `run` |

### 1.4 Scope Boundaries

**In scope:**
- Build-time auto-discovery and validation of OATF scenarios
- Compile-time embedding of valid YAML scenarios into the binary
- Scenario metadata extraction from OATF documents (`attack.id`, `attack.name`, `classification`)
- CLI commands for listing and inspecting built-in scenarios
- Integration with `run` via `--scenario` flag

**Out of scope:**
- User-defined scenario registration (use `--config` for custom configs)
- Runtime scenario downloading or updating
- OATF scenario authoring (see the OATF spec repository)

## 2. Functional Requirements

### F-001: Build-Time Scenario Discovery

The build script SHALL auto-discover and validate OATF scenarios from `scenarios/library/`.

**Acceptance Criteria:**
- `build.rs` walks `scenarios/library/**/*.yaml` recursively
- Each file is validated via `oatf::load()`; invalid documents are skipped with `cargo:warning`
- Valid scenarios are embedded via `include_str!` in generated code
- Metadata extracted from OATF documents: `attack.id` → name, `attack.name` → description, `classification.category` → category, `classification.tags` → tags
- If `scenarios/library/` does not exist, an empty registry is generated with a warning
- `cargo:rerun-if-changed=scenarios/library` ensures rebuilds when scenarios change

**Implementation:**

```rust
/// A built-in scenario embedded in the binary.
pub struct BuiltinScenario {
    /// Unique identifier derived from attack.id (lowercase, e.g., "oatf-001")
    pub name: &'static str,
    /// Short human-readable description from attack.name
    pub description: &'static str,
    /// Category mapped from attack.classification.category
    pub category: ScenarioCategory,
    /// Tags from attack.classification.tags
    pub tags: &'static [&'static str],
    /// Raw YAML content (embedded at compile time)
    pub yaml: &'static str,
}
```

**Registry (auto-generated by build.rs):**

```rust
/// Global registry — generated at build time, only valid OATF documents.
static BUILTIN_SCENARIOS: &[BuiltinScenario] =
    include!(concat!(env!("OUT_DIR"), "/builtin_scenarios.rs"));
```

### F-002: Scenario Inventory

The system SHALL embed scenarios from the official OATF scenario library (`github.com/oatf-spec/scenarios`).

**Setup (git submodule):**
```bash
# Clone with scenarios (--recursive fetches submodules)
git clone --recursive https://github.com/thoughtgate/thoughtjack

# Or if already cloned:
git submodule update --init

# Build with scenarios embedded
cargo build
```

**Category mapping (OATF → ThoughtJack):**

| OATF `classification.category` | `ScenarioCategory` |
|---|---|
| `capability_poisoning` | `Injection` |
| `response_fabrication` | `Injection` |
| `context_manipulation` | `Injection` |
| `temporal_manipulation` | `Temporal` |
| `availability_disruption` | `DoS` |
| `oversight_bypass` | `Protocol` |
| `cross_protocol_chain` | `MultiVector` |

### F-003: Server Run Integration

The system SHALL support running built-in scenarios via `scenarios run <name>`.

**Acceptance Criteria:**
- `scenarios run <name>` loads the embedded YAML directly
- All `run` execution flags work normally (`--mcp-server`, `--max-session`, etc.)
- Invalid scenario name produces error listing available scenarios

### F-004: Scenarios List Command

The system SHALL provide a command to list all available built-in scenarios.

**Acceptance Criteria:**
- `thoughtjack scenarios list` shows all built-in scenarios
- Supports filtering by category (`--category`) and tag (`--tag`)
- Supports JSON output (`--format json`) for tooling
- Shows name, category, and description
- Groups by category in human output

### F-005: Scenarios Show Command

The system SHALL provide a command to display the YAML content of a built-in scenario.

**Acceptance Criteria:**
- `thoughtjack scenarios show <name>` prints the raw YAML to stdout
- Output is the exact embedded YAML, unmodified
- Invalid scenario name produces error with suggestion

### F-006: Scenario Name Suggestion

The system SHALL suggest corrections for mistyped scenario names.

**Acceptance Criteria:**
- Suggest closest match using Damerau-Levenshtein distance
- Only suggest if distance ≤ 3
- Show suggestion in error message

## 3. Edge Cases

### EC-SCN-001: Unknown Scenario Name

**Scenario:** `thoughtjack scenarios run nonexistent`
**Expected:** Error listing available scenarios. If edit distance ≤ 3 from an existing name, include "Did you mean ...?" suggestion.

### EC-SCN-002: No Scenarios Library

**Scenario:** `cargo build` without `git submodule update --init`
**Expected:** Compiles with zero built-in scenarios and a `cargo:warning`. Commands like `scenarios list` return empty results.

### EC-SCN-003: Invalid OATF Document

**Scenario:** A YAML file in `scenarios/library/` fails `oatf::load()` validation
**Expected:** `cargo:warning=Skipping {path}: {error}` and the scenario is excluded from the binary.

### EC-SCN-009: Empty Category Filter

**Scenario:** `thoughtjack scenarios list --category protocol` when no protocol scenarios exist
**Expected:** Empty list with message: "No scenarios match the given filters."

### EC-SCN-010: Scenario List JSON Format

**Scenario:** `thoughtjack scenarios list --format json`
**Expected:** Valid JSON array. Empty array `[]` if no scenarios match filters.

### EC-SCN-013: Scenario Name Case Sensitivity

**Scenario:** `thoughtjack scenarios run OATF-001`
**Expected:** No match (names are lowercase). Suggestion offered: "Did you mean 'oatf-001'?"

## 4. Non-Functional Requirements

### NFR-001: Binary Size Impact

- Total embedded scenario YAML SHALL be < 500 KB
- Binary size increase SHALL be < 1% of total binary size

### NFR-002: Startup Performance

- Scenario lookup SHALL be O(n) on number of built-in scenarios (n < 50, negligible)
- No lazy parsing — scenarios parsed only when selected
- `scenarios list` SHALL complete in < 10ms (no config parsing involved)

### NFR-003: Maintainability

- Adding a new scenario requires only adding a YAML file to the OATF library repository and updating the submodule pin
- No Rust source changes required to add, remove, or update scenarios
- Scenarios validated at build time via `oatf::load()` — broken documents never ship

## 5. Build-Time Validation

Built-in scenarios are validated twice:

1. **Build time**: `build.rs` calls `oatf::load()` — invalid documents excluded
2. **Test time**: `all_scenarios_pass_oatf_validation` test re-validates all embedded YAML

```rust
#[test]
fn all_scenarios_pass_oatf_validation() {
    for scenario in list_scenarios(None, None) {
        let result = oatf::load(scenario.yaml);
        assert!(
            result.is_ok(),
            "Built-in scenario '{}' failed OATF validation: {:?}",
            scenario.name,
            result.err()
        );
    }
}
```
