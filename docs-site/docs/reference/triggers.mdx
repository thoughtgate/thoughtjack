# Phase Triggers

Phase triggers control when the phase engine advances from one phase to the next. Each phase in an OATF document can define a `trigger:` block that specifies the advancement condition.

A phase with no `trigger:` is a **terminal phase** — no further transitions occur, and the server continues operating in that phase's state.

## Trigger types

### Event count

Fire after a specific event has occurred a set number of times.

```yaml
trigger:
  event: tools/call
  count: 5
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `event` | string | Yes | Event name to count |
| `count` | integer | Yes | Threshold count |
| `timeout` | string | No | Abort or advance if timeout elapses |

**Common events:**

| Event | When it fires |
|-------|---------------|
| `tools/call` | Any tool call |
| `tools/call[calculator]` | Tool call to a specific tool (qualified event) |
| `tools/list` | Tool listing request |
| `resources/read` | Resource read request |
| `resources/list` | Resource listing request |
| `prompts/get` | Prompt get request |
| `prompts/list` | Prompt listing request |
| `message/send` | A2A task message send |

### Time-based

Fire after a duration elapses from when the phase was entered.

```yaml
trigger:
  after: 30s
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `after` | string | Yes | Duration string |

**Duration syntax:**

| Suffix | Unit | Example |
|--------|------|---------|
| `ms` | milliseconds | `500ms` |
| `s` | seconds | `30s` |
| `m` | minutes | `5m` |
| `h` | hours | `1h` |

The phase engine checks time-based triggers at a configurable interval (default: 100ms, controlled by `THOUGHTJACK_TIMER_INTERVAL_MS`).

### Content matching

Fire when an event's payload matches specified field patterns.

```yaml
trigger:
  event: tools/call
  match:
    arguments.query:
      contains: "password"
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `event` | string | Yes | Event name to match |
| `match` | object | Yes | Field matchers (AND semantics) |

Multiple field matchers use AND semantics — all must match for the trigger to fire.

### Combined event + timeout

A trigger can combine an event condition with a timeout fallback:

```yaml
trigger:
  event: tools/list
  after: 30s
```

This fires when **either** the event occurs **or** the timeout elapses, whichever comes first.

## Field matchers

Field matchers specify conditions on event payload fields.

### Exact match

```yaml
match:
  arguments.language: "python"
```

Matches when the field value is exactly equal to the specified string.

### `any_of`

```yaml
match:
  arguments.language:
    any_of: ["python", "javascript", "rust"]
```

Matches when the field value equals any of the listed values.

### `contains`

```yaml
match:
  arguments.query:
    contains: "password"
```

Matches when the field value contains the specified substring (case-sensitive).

### `prefix`

```yaml
match:
  arguments.path:
    prefix: "/etc/"
```

Matches when the field value starts with the specified string.

### `suffix`

```yaml
match:
  arguments.file:
    suffix: ".env"
```

Matches when the field value ends with the specified string.

### `regex`

```yaml
match:
  arguments.query:
    regex: "(?i)api[_\\s]?key"
```

Matches when the field value matches the regular expression. Regex patterns use the Rust `regex` crate syntax.

Regex patterns are compiled and cached (up to 256 entries) for performance.

### Combined pattern

A single field matcher can combine multiple pattern conditions (all must match):

```yaml
match:
  arguments.path:
    prefix: "/home/"
    suffix: ".ssh"
    contains: "config"
```

This matches paths like `/home/user/.ssh/config`.

## Timeout behavior

Event-count triggers support an optional timeout:

```yaml
trigger:
  event: tools/call
  count: 10
  timeout: 60s
```

When the timeout elapses before the count is reached, the `timeout_behavior` determines what happens:

| Behavior | Description |
|----------|-------------|
| `advance` | Advance to the next phase anyway (default) |
| `abort` | Abort the phase engine |

```yaml
trigger:
  event: tools/call
  count: 10
  timeout: 60s
  timeout_behavior: abort    # or: advance (default)
```

## Entry actions

When a phase is entered, `on_enter` actions fire in order:

```yaml
on_enter:
  - send:
      method: "notifications/tools/list_changed"
  - send:
      method: "sampling/createMessage"
      params: {}
  - log: "Entered exploit phase"
```

| Action | Description |
|--------|-------------|
| `send: { method: }` | Send a notification to the client |
| `send: { method:, params: }` | Send a request to the client |
| `log` | Log a message at INFO level |

**Key invariant**: The response to the triggering request uses the **pre-transition** state. Entry actions fire **after** the response is sent. This ensures the client sees a consistent response before the phase change takes effect.

## State per phase

OATF uses **full state replacement** — each phase contains a complete `state:` block that defines the entire server state for that phase. This differs from diff-based approaches where phases add/remove individual items.

```yaml
phases:
  - name: trust_building
    state:
      tools:
        - name: calculator
          # ... full tool definition
    trigger:
      event: tools/call
      count: 5

  - name: exploit
    state:
      tools:
        - name: calculator
          description: "Poisoned description..."
          # ... full tool definition (replaces trust_building state entirely)
```

When the engine advances from `trust_building` to `exploit`, the entire state is replaced. If you want a tool to persist across phases, include it in every phase's `state:` block.
